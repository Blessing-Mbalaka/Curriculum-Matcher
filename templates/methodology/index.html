CurriculumMatch methodology

A normalized embedding workflow for comparing curriculum content with live labour-market demand.

Limitations and Governance

Human review required

CurriculumMatch is a decision-support system, not an automatic curriculum authority. Its outputs depend on the quality of job adverts, course text, extracted skills, and human review decisions.

Skill extraction is evidence-bound The extractor can miss skills when job adverts use unusual phrasing, abbreviations, broad responsibilities, or implicit requirements. Thin descriptions can understate real market demand.
LLM verification suggests, not decides Ollama suggestions are stored as candidates. They should be inspected against the source evidence before approval because language models can overgeneralise or propose labels that are too broad.
Human approval shapes future learning Reviewed skills feed the dynamic lexicon and retraining datasets. Consistent review standards matter: approving noisy labels can make future extraction noisier.
Embeddings capture similarity, not equivalence A high semantic score means two texts are close in language and evidence, but it does not prove that learning outcomes, depth, assessment, or graduate competence are equivalent.
Forecasts are exploratory Skill forecasts use lightweight trends from filtered evidence counts. They are useful for scanning direction, but not for formal labour-market prediction.
Dataset coverage can bias results Adzuna/API coverage, uploaded CSVs, duplicated adverts, regional filters, and institutional course text availability all influence what the system sees and ranks.
Input layer

Curriculum evidence

Each course contains modules. Module content can be pasted directly or parsed from PDF, Word, text, and PowerPoint files. The course scraper can also import programme/module evidence from relevant pages.

Market layer

Job-market evidence

Jobs enter through CSV upload, manual entry, or Adzuna. Adzuna duplicates are managed with external IDs and content fingerprints so repeated listings do not distort market demand.

Processing layer

Analysis runs

A run snapshots current courses and jobs, vectorises content, extracts skills, stores match rows, and records task progress so visualisations can preserve the last usable result while new work is running.

Preprocessing Boundary

Natural text preserved

Course material is lightly cleaned but not aggressively stripped before BERT, spaCy, or semantic embedding. The system preserves natural sentence context because words such as with, for, in, to, and and help contextual models understand whether a phrase is a skill, an outcome, a prerequisite, or general description.

BERT and embeddings These models receive natural course text with whitespace, bullet, line-break, and obvious extraction-noise cleanup only. Stop-word removal is avoided because it can damage sentence meaning and weaken contextual skill detection.
Rule-based fallback Lowercasing, punctuation cleanup, and token normalization are used only for phrase matching, regex matching, aliases, and Word2Vec fallback tokenization.
Why separate paths? The semantic path needs readable text; the exact-match path needs normalized tokens. Keeping both paths separate improves recall without throwing away context.

NER Skill Evidence

Reviewable records

Jobs and course modules store both simple skill lists and richer NER skill entities. Each entity can include a stable skill ID, label, tier, skill type, source method, confidence, mention count, pattern, position signature, extracted text, and review status. The extractor now loads a fine-tuned DistilBERT token-classification model as a contextual NER layer when models/bert_skill_ner is present.

BERT skill NER BERT reads each token in sentence context, so it can learn skill mentions that do not exactly match a fixed phrase list. Its predictions are confidence-thresholded and combined with spaCy, phrase matching, aliases, and regex evidence.

How We Classify Skills

Simple taxonomy

In this system, a skill is any reusable capability, tool, method, or field-specific competence that a person can apply in work or study. We avoid treating every noun as a skill: the term must be useful for comparing what jobs ask for with what courses teach.

Technical skills These are tools, systems, data methods, software, platforms, or technical practices. Examples include Python, SQL, Power BI, cloud, automation, reporting, dashboards, software development, and machine learning.
Soft skills These are transferable human capabilities used across many roles. Examples include communication, leadership, teamwork, negotiation, presentation, stakeholder relations, problem solving, and time management.
Business skills These are management, commercial, finance, HR, operations, governance, risk, marketing, sales, compliance, procurement, and strategy capabilities that are especially relevant to business-school curricula.
Domain skills These are specialised sector or subject-area phrases that look skill-like but do not clearly fall into technical, soft, or business buckets. They are kept as domain skills so a human reviewer can decide whether to refine them later.
Why the labels are reviewable The first classification is rule-assisted: the system scores words in the skill phrase and nearby context against transparent keyword groups. A human can change the type during review, and that reviewed decision becomes the stronger learning signal.
Classification rule in plain English
Look at the skill phrase + evidence context. Count signals for technical, soft, and business categories. If no category is strong enough, keep it as domain. Then assign a tier such as tool, method, transferable, specialised, or candidate.

Normalized Embeddings

Semantic signal

The preferred backend is a sentence-transformer model. It encodes text with normalized embeddings, meaning each vector is scaled to unit length before comparison. When that model is unavailable, the system falls back to Word2Vec by averaging token vectors from the current corpus.

Sentence-transformer encode(text, normalize_embeddings=True) creates a unit-length vector for each module and job advert.
Word2Vec fallback Text is lowercased, tokenised, trained with a seeded Word2Vec model, then represented as the mean of known token vectors.
Why normalize? Normalization makes the comparison focus on direction, or meaning, rather than document length or vector magnitude.

AI Model Ensemble

Ranking layer

The ranking layer combines several models instead of relying on a single signal. Semantic embeddings measure meaning, BERT and spaCy provide skill evidence, extraction confidence measures reliability, and a decision-tree scorer applies interpretable threshold rules for strong, medium, and weak matches.

Decision tree The current tree is rule-based and transparent: high semantic similarity, strong skill coverage, and high extraction confidence produce the strongest rank boost.
Future supervised model When enough reviewed outcomes exist, the same features can train a scikit-learn decision tree, random forest, or gradient boosting model without changing the stored evidence shape.
Stored explanation Each match can store semantic, skill coverage, confidence, decision-tree, and final ensemble scores in the score breakdown field.
Semantic similarity high? Start with meaning, then check evidence strength.
Coverage + confidence strong Boost as a likely curriculum match.
Evidence weak Keep visible, but rank below stronger matches.

Confidence Ranking

Evidence quality

Confidence is calculated from matched skill entities found in both course and job evidence. High-confidence BERT, spaCy, and phrase evidence increases the final rank; weak or missing skill evidence limits how far a semantic match can rise.

Confidence score
mean matched skill confidence across course evidence and job evidence

Why this matters
Semantic similarity can find related meaning; confidence helps separate strong evidence from accidental wording overlap.

School Skill Matrix Calculations

Evidence counts

The School Skill Matrix is calculated from the currently selected result set. A skill is first refined into a business-school label, then counted across course-to-job comparisons.

\[ D_s = C_s + G_s \]
Demand evidence \(D_s\) equals covered evidence \(C_s\) plus gap evidence \(G_s\) for skill \(s\).
\[ \operatorname{GapPercent}_s = \frac{G_s}{\max(1, D_s)} \times 100 \]
Gap % shows the share of demand evidence where the skill is missing from the matched course.
\[ \operatorname{CoveragePercent}_s = \frac{C_s}{\max(1, D_s)} \times 100 \]
Coverage % shows the share of demand evidence where the course already contains that skill.
\[ \operatorname{GapBarWidth}_s = \frac{G_s}{\max_j(G_j)} \times 100 \]
Bar width is relative to the largest missing-evidence skill in the selected school or current filter.

Skill Forecasting

Time series

The Data Export forecast button groups filtered skill evidence by extraction year. For jobs, the preferred date is the job posting date; otherwise the system uses the stored creation timestamp. For course modules and uploaded documents, the module/document storage timestamp anchors the year.

\[ y_{s,t} = \operatorname{count}(\operatorname{skill}=s, \operatorname{year}=t) \]
For each top skill \(s\), the chart plots observed yearly chunk counts \(y_{s,t}\).
\[ \hat{y}_{s,t+1} = \max(0, y_{s,t} + \operatorname{slope}(y_s)) \]
The one-year projection is a lightweight trend forecast designed for visual exploration, not a statistical guarantee.

How To Read the Outputs

Analyst guide
High score The module language and extracted skills are close to the selected job advert. Inspect matched skills to confirm the alignment is educationally meaningful.
Low score The course may lack relevant market language, the job may be outside the curriculum scope, or the text may be too thin for reliable embeddings.
Missing skills Skills found in job adverts but not in the course modules. These are candidate curriculum gaps, not automatic curriculum changes.
Extra skills Skills taught by modules but not demanded in a particular job. These can still be valuable, especially for broader graduate outcomes.
Network edge width Thicker edges represent stronger similarity percentages. The layout is static so repeated refreshes do not disrupt interpretation.
Cross-tab heatmap Rows are courses, columns are job adverts, and cell color is the cosine similarity percentage for that course-to-job edge.
Forecast line Solid segments show observed yearly skill evidence. Dotted segments show the next-year projection from the current filtered evidence.
Skill clusters Nearby skills share terms, sector context, skill type, or repeated co-occurrence. Use clusters as a navigation aid before reviewing individual skill evidence.
Duplicate control

Adzuna deduplication

The app checks Adzuna external IDs and a normalized content fingerprint built from title, company, recruiter, location, reference, and description. Duplicate jobs are skipped and counted in task notes.

Reliability control

Live task heartbeat

Background tasks update a heartbeat timestamp. If a dev server restart kills a thread, stale tasks are marked clearly instead of appearing to process forever.

Visual control

Preserved visuals

Dashboard visuals only update when new visual data exists. During processing, the last usable results remain visible so analysis does not disappear mid-run.
